# Paysage du problème

## Problème principal
Le projet part d’un constat clair : des porteurs de projet peu expérimentés en cadrage produit et technique utilisent des LLM pour aller très vite dans la création d’un SaaS, mais ces modèles improvisent trop facilement la stack, l’ordre des décisions, la structure documentaire et parfois même la logique produit.

Le problème n’est donc pas seulement “l’IA code mal”.
Le problème est plus précis :

- l’intention initiale se déforme au fil des prompts ;
- les décisions importantes ne sont pas suffisamment justifiées ni mémorisées ;
- la documentation produite est fragmentée, incomplète ou contradictoire ;
- le projet paraît avancer, mais ses fondations restent fragiles.

## Contexte d’apparition
Le problème apparaît surtout dans ce contexte :

- un utilisateur arrive avec une idée de SaaS encore partiellement floue ;
- il s’appuie sur des outils de build IA pour accélérer ;
- l’outil commence à produire vite ;
- la vitesse donne une impression de maîtrise ;
- mais la cohérence d’ensemble n’est pas réellement garantie.

Le moment critique n’est pas le premier écran généré.
Le moment critique est le passage d’une idée séduisante à un produit sérieux, maintenable, transmissible et extensible.

## Personnes ou organisations touchées
### Cible principale
- vibecodeurs débutants dans l’entrepreneuriat ;
- fondateurs non techniques ou semi-techniques ;
- porteurs de projet qui savent prompter mais ne savent pas encore cadrer proprement un produit.

### Cibles secondaires plausibles
- freelances ou petites agences qui veulent produire plus vite sans dérive ;
- PM ou designers qui prototypent fortement avec l’IA ;
- équipes très petites qui n’ont pas encore de méthode documentaire robuste.

## Fréquence supposée
### Confirmé
- Le problème est structurel dans le brief : il ne dépend pas d’un cas isolé mais d’un mode de travail entier basé sur les LLM.

### Hypothèses de travail
- Le problème revient souvent chez les débutants qui passent vite de l’idée au build.
- La fréquence augmente avec la popularité des outils de génération d’apps et d’agents de code.
- Le problème est plus fort dès que le projet dépasse un simple prototype jetable.

### Inconnu
- La fréquence réelle par segment précis :
  - solo founder non-tech ;
  - builder technique léger ;
  - agence ;
  - équipe produit.

## Gravité supposée
La gravité paraît élevée pour trois raisons :

1. le dommage n’est pas seulement esthétique ;
2. il touche des décisions coûteuses à corriger ensuite ;
3. il crée un faux sentiment de progression.

### Conséquences graves plausibles
- mauvaise stack retenue trop tôt ;
- périmètre mal pensé ;
- dette de cohérence ;
- réécriture ou redémarrage partiel du projet ;
- difficulté à transmettre le projet à une équipe humaine ou à un autre agent ;
- perte de confiance du porteur de projet.

## Comment le problème est géré aujourd’hui
Les contournements actuels décrits dans le brief sont déjà cohérents :
- bricolage manuel de prompts ;
- usage de plusieurs IA sans mémoire consolidée ;
- accumulation de docs séparés ;
- correction tardive après dérive ;
- redémarrage du projet quand le système devient trop confus.

À cela s’ajoutent des comportements de marché plausibles :
- utilisation directe d’outils “idea-to-app” pour générer vite ;
- usage de docs collaboratifs ou PRD templates pour remettre de l’ordre ;
- recours à un humain expérimenté quand le projet devient trop risqué.

## Pourquoi ce problème mérite probablement d’être attaqué
L’opportunité n’existe que si l’on accepte cette nuance :

- le marché semble déjà très actif sur la génération rapide d’applications ;
- mais la douleur de cohérence, de cadrage et de transmissibilité paraît moins bien couverte ;
- le projet peut donc avoir une place s’il ne se présente pas comme “encore un builder IA”, mais comme une couche de structuration et de fiabilisation.

## Confirmé
- Le point d’entrée retenu est le SaaS.
- La cible principale actuelle est le vibecodeur débutant avec ambition entrepreneuriale.
- Le problème principal visé est l’improvisation des LLM sur la stack, le déroulé et certaines décisions structurantes.
- Les conséquences perçues comme critiques sont la mauvaise stack, le mauvais cadrage et la faiblesse documentaire.
- Le livrable le plus précieux visé est un dossier d’exécution consolidé exploitable par humains et LLM.

## Hypothèses de travail
- La douleur est suffisamment forte pour justifier un produit dédié.
  - Impact : très fort sur la désirabilité.
  - Pourquoi cette hypothèse a été retenue : elle est cohérente avec la montée rapide des usages de build IA et avec la difficulté connue à maintenir de la cohérence sur des projets plus sérieux.

- La douleur augmente fortement avec la complexité du projet.
  - Impact : fort sur le positionnement.
  - Pourquoi cette hypothèse a été retenue : plus le projet devient durable, plus l’absence de mémoire de décision et de structure documentaire coûte cher.

- La valeur du projet sera perçue davantage comme réduction du risque que comme simple gain de vitesse.
  - Impact : fort sur le messaging, le pricing et la preuve de valeur.
  - Pourquoi cette hypothèse a été retenue : la promesse la plus distinctive n’est pas “aller plus vite”, mais “éviter une dérive coûteuse”.

## Inconnus
- Quel segment ressent la douleur la plus forte et paie le plus volontiers.
  - Pourquoi ce point reste inconnu : la cible est encore large.
  - Quel impact potentiel : très fort sur le go-to-market.

- À partir de quel niveau de complexité le besoin devient vraiment aigu.
  - Pourquoi ce point reste inconnu : aucune frontière claire n’a encore été validée.
  - Quel impact potentiel : fort sur la définition du périmètre initial.

- Quelle partie de la douleur domine réellement à l’achat :
  - mauvaise stack ;
  - incohérence documentaire ;
  - perte de temps ;
  - perte de confiance ;
  - difficulté à piloter le build.
  - Pourquoi ce point reste inconnu : le brief agrège plusieurs douleurs voisines.
  - Quel impact potentiel : très fort sur la proposition de valeur.

## Bloquants
- Aucun bloquant absolu pour reconnaître l’existence potentielle du problème.
- En revanche, il reste bloquant pour la suite de ne pas savoir :
  - quel segment souffre le plus ;
  - quelle douleur déclenche vraiment l’achat ;
  - à partir de quel niveau de complexité le produit devient indispensable.
