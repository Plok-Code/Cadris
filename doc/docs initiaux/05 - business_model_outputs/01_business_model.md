# 01_business_model.md

## Resume
Le modele economique le plus plausible a ce stade consiste a vendre **une capacite de cadrage multi-agent de niveau entreprise**, materialisee par un corpus documentaire serieux et un dossier d'execution consolide.

Le produit ne vend pas seulement :
- une source de verite ;
- un export ;
- un generateur de docs ;
- un builder d'app.

Il vend la capacite a faire travailler, autour d'un projet, des agents representant les vrais domaines d'une entreprise :
- strategie ;
- produit ;
- UX/UI ;
- technique ;
- data ;
- legal si necessaire ;
- execution / go-to-market.

## Qui paie

### Payeur principal recommande
- solo founder ambitieux ;
- indie builder avance ;
- micro-equipe early-stage ;
- petite structure qui veut cadrer proprement avant de build plus loin.

### Pourquoi ce payeur est plausible
- il ressent directement le cout des mauvaises decisions ;
- il a besoin d'aller vite sans improvisation ;
- il n'a pas forcement toutes les expertises sous la main ;
- il peut percevoir de la valeur dans un "comite d'experts agentiques" plus que dans un simple document.

### Acheteur / utilisateur / decideur
En V1, ces roles restent souvent confondus.
Mais le produit commence a etre vendable aussi comme outillage d'une micro-equipe, pas seulement d'un individu.

## Ce qui est vendu

### Ce qui ne doit pas etre vendu
- un simple app builder ;
- une promesse de generation magique ;
- un outil de documentation generique ;
- une prestation de conseil deguisee ;
- un simple "workspace de memoire".

### Ce qui est vendu en priorite
Une **organisation de cadrage multi-agent** qui :
- questionne ;
- detecte les angles morts ;
- croise les points de vue metier ;
- structure les arbitrages ;
- produit les documents necessaires ;
- livre un dossier d'execution coherent.

### Materialisation concrete de la valeur
La valeur doit se materialiser en deux niveaux :
1. **la machine de cadrage** : les agents qui cooperent et challengent le projet ;
2. **le resultat** : les documents et le dossier d'execution consolide.

## Pourquoi cela a de la valeur
La valeur percue vient surtout de 5 leviers :
1. reduction du risque de mauvaise decision structurante ;
2. reduction de la derive entre domaines ;
3. gain de temps par rapport a un cadrage manuel disperse ;
4. professionnalisation du projet avant le build ;
5. capacite a lancer un build avec un niveau de clarte bien superieur.

## Logique generale du modele economique

### Mecanique recommandee
- point d'entree : un projet ou une idee a cadrer serieusement ;
- coeur de valeur : une boucle de travail multi-agent ;
- actif de sortie : le dossier d'execution consolide ;
- extension naturelle : mises a jour, pivots, nouvelles missions, build-review aval.

### Lecture business
- la facade vendable n'est pas seulement le dossier ;
- le dossier prouve la valeur d'un **systeme d'experts agentiques coordonnes**.

## Recommandation de modele economique

### Recommandation principale
**Produit logiciel a logique hybride :**
- une entree ancree sur une mission ou un projet ;
- une profondeur de service differenciee selon la couverture agentique et documentaire ;
- une continuite payante si le projet continue a etre cadre, revise ou supervise.

### Pourquoi ce modele est coherent
- il colle mieux a ton vrai produit qu'un simple "document as a service" ;
- il laisse monetiser la profondeur de travail, pas seulement le fichier final ;
- il reste comprehensible si le packaging reste simple.

## Ce qui reste speculatif
- quelle part de la valeur percue vient de la machine agentique elle-meme vs du dossier final ;
- quel niveau de couverture documentaire les utilisateurs sont prets a payer tout de suite ;
- quel packaging convertit le mieux : mission, projet, abonnement, ou hybride ;
- quelle montee en gamme est legitime entre cadrage et supervision du build.
