# 03_revenue_model.md

## Type de revenu recommandé
### Type principal
Revenu logiciel récurrent.

### Type secondaire plausible
Complément transactionnel ou one-shot lié :
- à l’activation d’un nouveau projet ;
- à un export / handoff avancé ;
- à une phase kickoff plus poussée.

## Fréquence
### Recommandation principale
Mensuelle au départ.

### Pourquoi
- adaptée à une cible solo / micro-équipe ;
- rythme naturel pour un projet en construction ;
- plus simple à comprendre qu’un modèle annuel ou fortement usage-based.

## Unité de valeur
### Unité de valeur principale
**Un projet SaaS actif tenu sous contrôle.**

### Sous-unités de valeur
- source de vérité fiable ;
- décisions stabilisées ;
- capacité à reprendre ;
- capacité à transmettre ;
- cohérence meilleure entre intention et exécution.

## Unité de facturation
### Recommandation
**Projet actif par mois** comme unité d’ancrage.

### Pourquoi
- “projet” rend la valeur concrète ;
- “par mois” soutient la continuité ;
- évite l’effet taxi meter.

## Logiques d’extension possibles
### Extension 1 — Projets supplémentaires
Montée en gamme naturelle quand un builder porte plusieurs produits.

### Extension 2 — Collaboration / handoff
Montée en gamme plausible dès qu’un autre humain doit reprendre ou comprendre le projet.

### Extension 3 — Exports avancés / mémoire renforcée
Montée en gamme cohérente si le livrable devient une vraie pièce de travail pour IDE / repo / agents.

### Extension 4 — Offre plus premium
Possible plus tard pour :
- micro-équipes ;
- cas semi-conciergerie ;
- onboarding plus accompagné.

## Ce qui paraît moins adapté en V1
- seat-based pur comme modèle principal ;
- crédits visibles comme promesse centrale ;
- setup fee obligatoire ;
- freemium trop large sans preuve de conversion.

## Lecture stratégique
Le revenu doit partir d’un objet acheté simple :
**“je sécurise ce projet”**
puis évoluer vers :
**“je garde ce projet propre et transmissible dans la durée.”**
