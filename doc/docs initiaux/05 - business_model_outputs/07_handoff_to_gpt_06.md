# 07_handoff_to_gpt_06.md

## Résumé exécutif
Le projet paraît économiquement crédible s’il est vendu comme **source de vérité IA-ready pour builders déjà un peu avancés**, au moment où un proto ou une petite app commence à dériver. Le produit ne doit pas être présenté comme un simple app builder ni comme un générateur de documentation abstraite. La valeur monétisable la plus plausible réside dans la réduction du risque de rebuild, la continuité de décision, la reprise plus rapide, et la transmission du projet.

## Valeur monétisable identifiée
- remise au carré d’un projet déjà réel ;
- dossier canonique exportable ;
- réduction de dérive entre idées, décisions et exécution ;
- meilleure reprise du projet après pause ;
- transmission plus propre à un humain ou à un agent ;
- professionnalisation du passage proto -> vrai SaaS.

## Modèle de revenu recommandé
### Recommandation principale
Logiciel récurrent ancré sur le **projet actif**, avec promesse :
“je garde ce projet clair, transmissible et exploitable.”

### Forme économique recommandée
- base récurrente simple ;
- possibilité d’entrée ou d’activation liée au projet ;
- montée en gamme par projets supplémentaires, profondeur, collaboration et handoff.

## Hypothèses de pricing
### Hypothèse principale
Abonnement mensuel avec 1 projet actif inclus.

### Alternatives plausibles
- paiement par projet pour une V1 ultra simple ;
- kickoff / export initial + abonnement de continuité ;
- paliers ensuite selon profondeur, nombre de projets, et collaboration.

### Ce qu’il faut éviter
- crédits visibles comme unité principale ;
- pricing au run ;
- narration “documentation” seule ;
- modèle trop service au départ.

## Hypothèses sensibles
- la promesse “réduction du risque de dérive” convertit mieux que “documentation structurée” ;
- le dossier canonique est le bon objet concret de vente initiale ;
- la continuité / mémoire crée suffisamment de rétention pour soutenir du récurrent ;
- le builder post-proto chaotique est plus acheteur que le pur débutant.

## Points confirmés
- le produit ne doit pas être compris comme un simple app builder ;
- la cible connaît déjà un peu les outils IA ;
- la douleur apparaît surtout quand le projet grossit ;
- les jobs les plus solides tournent autour de l’ordre, de la source de vérité et de la transmission ;
- le contexte d’achat le plus crédible n’est pas l’idée pure, mais le projet déjà existant qui commence à compter.

## Hypothèses de travail
- le meilleur ICP de départ est le solo founder / indie builder / micro-équipe déjà capable de sortir une petite app ;
- le bon wedge initial est un petit SaaS B2B sérieux plutôt qu’un périmètre “tout SaaS” ;
- la façade vendable du produit est le dossier canonique exportable ;
- la profondeur économique à moyen terme vient de la continuité.

## Inconnus
- phrase de douleur qui convertit le mieux ;
- wedge SaaS exact à inclure en V1 ;
- moment exact du paiement ;
- part respective de valeur entre export ponctuel et maintien continu ;
- corridor de prix acceptable ;
- niveau de profondeur documentaire optimal.

## Bloquants
- formuler la douleur d’achat principale ;
- choisir les cas inclus / exclus du wedge ;
- trancher la frontière avant-code vs pendant build ;
- valider le segment post-Lovable ;
- valider l’unité de valeur préférée par les prospects.

## Niveau de fiabilité
**Fiabilité moyenne à bonne.**

### Pourquoi
- le cadrage cible / douleur / valeur tient déjà ;
- la structure de revenu plausible est claire ;
- le pricing final reste encore dépendant de validations terrain.

## Ce que le GPT 06 doit vérifier en priorité
1. La formulation d’offre la plus achetable.
2. Le messaging principal : réduire le rebuild, garder une source de vérité, ou professionnaliser le proto.
3. Le wedge SaaS V1 exact.
4. Le bon objet de vente apparent : dossier, continuité, ou hybride.
5. L’unité de pricing la plus intelligible sur landing page.
6. Les preuves minimales à obtenir pour fixer un corridor de prix.
