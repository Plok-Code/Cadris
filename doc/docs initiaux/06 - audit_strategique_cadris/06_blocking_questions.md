# Questions bloquantes

## Objet
Ce document ne signifie pas que le projet est arrêté.
Il liste les questions qui bloquent un passage vers :
- un messaging net ;
- un packaging économique fiable ;
- un pricing défendable ;
- une V1 très bien cadrée.

## 1. Quelle phrase décrit le mieux la douleur d’achat principale ?
### Pourquoi cela bloque
Le produit peut être acheté pour :
- éviter un rebuild ;
- arrêter la dérive ;
- obtenir une vraie source de vérité ;
- professionnaliser un proto ;
- transmettre sans réexpliquer tout le contexte.

Sans hiérarchie claire, le message reste partiellement flou.

### Ce qu’il faut obtenir
- 1 phrase principale ;
- 1 à 2 douleurs secondaires maximum ;
- le vocabulaire exact de l’utilisateur.

### Statut
**Bloquant**

## 2. Quel wedge SaaS exact sert-on en premier ?
### Pourquoi cela bloque
“Vrai SaaS” reste trop large.
Le projet a besoin d’un périmètre plus concret pour :
- la landing ;
- l’onboarding ;
- les exemples ;
- les exclusions ;
- la discipline produit.

### Ce qu’il faut obtenir
- 3 cas explicitement inclus ;
- 3 cas explicitement exclus ;
- un niveau de complexité acceptable ;
- un angle simple de type micro-SaaS B2B mono-workflow ou équivalent.

### Statut
**Bloquant**

## 3. Le produit vend-il surtout un dossier exportable ou une continuité vivante ?
### Pourquoi cela bloque
Le business model, le pricing, le messaging et la roadmap changent selon que l’on vend :
- un livrable concret ;
- une mémoire continue ;
- un système hybride.

### Ce qu’il faut obtenir
- la forme de valeur la plus vite comprise ;
- la forme de valeur la plus vite achetée ;
- la forme de valeur la plus retenue.

### Statut
**Bloquant**

## 4. Le produit intervient-il avant le code, juste après le proto, ou aussi pendant l’exécution ?
### Pourquoi cela bloque
Le moment d’entrée change :
- la promesse ;
- la structure de revenu ;
- la charge support ;
- l’expérience produit.

Les documents récents convergent vers :
- avant-code fort ;
- pendant-build léger ;
mais il faut encore le verrouiller dans le discours d’offre.

### Ce qu’il faut obtenir
- un point d’entrée principal ;
- un point d’entrée secondaire ;
- ce que le produit ne promet pas encore.

### Statut
**Bloquant**

## 5. Le sous-segment post-Lovable / post-builder bloqué est-il un vrai segment acheteur ?
### Pourquoi cela bloque
Le segment est narrativement séduisant.
Mais il peut n’être qu’un angle de wording.

### Ce qu’il faut obtenir
- entretiens directs ;
- motifs de frustration récurrents ;
- capacité réelle à payer ;
- signal qu’ils achètent autre chose qu’une migration improvisée.

### Statut
**Bloquant**

## 6. Quel niveau de profondeur documentaire est suffisamment puissant sans devenir lourd ?
### Pourquoi cela bloque
Trop peu de profondeur affaiblit la valeur.
Trop de profondeur ralentit l’achat, l’usage et augmente le support.

### Ce qu’il faut obtenir
- le minimum crédible ;
- le niveau premium ;
- le format préféré : dossier, mémoire, export, checklist de décisions, autre.

### Statut
**Bloquant**

## 7. À quelle fréquence les utilisateurs reviennent-ils réellement ?
### Pourquoi cela bloque
Le modèle économique récent s’oriente vers :
- projet actif ;
- continuité ;
- possible récurrent.

Mais un besoin trop ponctuel fragiliserait un abonnement mensuel pur.

### Ce qu’il faut obtenir
- fréquence des revisites ;
- moments de retour naturels ;
- répartition entre usage initial intense et usage continu léger.

### Statut
**Bloquant pour fixer le packaging économique final**

## 8. Quel est le coût moyen d’un client actif ?
### Pourquoi cela bloque
Les unit economics signalent :
- coût IA variable ;
- support potentiellement lourd ;
- usage non uniforme ;
- risque d’un flat trop généreux.

### Ce qu’il faut obtenir
- coût moyen par projet actif ;
- variance entre petits et gros utilisateurs ;
- intensité des fonctions les plus coûteuses ;
- volume de support par client actif.

### Statut
**Bloquant pour fixer un pricing public fiable**

## 9. Quel niveau de bornage est nécessaire si l’on garde une logique mensuelle ?
### Pourquoi cela bloque
Sans garde-fous, un mensuel flat peut devenir :
- incompris ;
- peu rentable ;
- ou désaligné avec la valeur.

### Ce qu’il faut obtenir
- limites de profondeur ;
- limites de projets ;
- limites de collaboration ;
- limites de support implicite ;
- limites des exports avancés.

### Statut
**Bloquant pour la tarification finale**

## 10. Quelle doctrine économique doit être tenue officiellement ?
### Pourquoi cela bloque
Le projet a encore des traces d’un ancien modèle crédits dans certains documents amont.
La doctrine récente est différente.

### Ce qu’il faut obtenir
- un arbitrage officiel :
  - crédits = historique ;
  - projet actif / continuité = modèle cible ;
- une formulation unique pour éviter toute ambiguïté.

### Statut
**Bloquant de cohérence interne**

## Priorité de résolution
1. douleur principale ;
2. wedge exact ;
3. objet vendu exact ;
4. moment d’entrée produit ;
5. profondeur documentaire minimale ;
6. doctrine économique officielle ;
7. fréquence de revisite ;
8. coût moyen / support moyen ;
9. bornage du mensuel.

## Lecture finale
Ces questions ne bloquent pas le droit de transmettre.
Elles bloquent surtout :
- un message public verrouillé ;
- un pricing final public ;
- une promesse commerciale très précise.
