# Audit de cohérence stratégique

## Périmètre de l’audit
Cet audit porte sur la cohérence entre :
- le problème principal ;
- la cible prioritaire ;
- les JTBD principaux ;
- la proposition de valeur ;
- le modèle économique ;
- le périmètre V1 ;
- les hypothèses de pricing et d’unit economics.

Il ne cherche pas à créer une nouvelle stratégie complète.  
Il cherche à vérifier si les fondations actuelles sont assez cohérentes pour transmettre le projet à l’étape suivante.

## Résumé de la situation
Le projet Cadris cherche à aider un builder IA déjà un peu avancé à transformer un proto brouillon en base SaaS claire, exploitable et transmissible, puis à garder la cohérence du projet pendant la suite du build.

Le cœur de la douleur décrit dans les documents est stable :
- les LLM font avancer le projet vite ;
- mais ils laissent souvent une stack, une documentation et une logique produit fragiles ;
- dès que la complexité augmente, l’utilisateur ne sait plus ce qui fait foi ;
- modifier, reprendre, transmettre ou lancer devient risqué.

Le cœur de cible a aussi gagné en netteté :
- pas le novice complet ;
- plutôt le solo founder, l’indie builder ou la micro-équipe early-stage ;
- déjà capables de sortir une petite app avec l’IA ;
- mais bloqués quand le projet doit devenir modifiable, transmissible et durable.

Le projet n’est donc plus présenté comme :
- un app builder générique ;
- un simple générateur de documentation ;
- ni un consultant déguisé.

Il est présenté comme :
- une source de vérité IA-ready ;
- un dossier canonique exportable ;
- puis une continuité légère de décision pendant le build.

## Problème principal audité
### Formulation retenue
Le problème principal n’est pas “les gens manquent de documents”.
Le problème principal est :

**un proto IA peut sembler avancer, puis devenir opaque, fragile et difficile à modifier ou transmettre quand la complexité monte.**

### Niveau de robustesse
Ce point est solide parce qu’il est cohérent dans :
- la vision ;
- la proposition de valeur ;
- les JTBD ;
- le PRD ;
- le business model.

### Limite restante
La douleur existe clairement.
Ce qui reste moins verrouillé, c’est sa formulation d’achat exacte sur le marché :
- “éviter le rebuild” ;
- “arrêter la dérive” ;
- “retrouver une source de vérité” ;
- “professionnaliser le proto”.

## Cible principale auditée
### Cible retenue
La cible la plus cohérente est :
- builder déjà initié aux outils IA ;
- solo founder / indie builder / micro-équipe early-stage ;
- projet déjà commencé ;
- besoin de sortir d’un proto fragile vers un vrai SaaS plus tenable.

### Pourquoi cette cible est cohérente
Cette cible :
- ressent directement le problème ;
- subit elle-même le coût de dérive ;
- décide souvent seule ;
- peut acheter en self-serve ou achat léger ;
- a un besoin concret de reprise, de transmission et de continuité.

### Fragilité restante
La cible reste encore large si elle n’est pas découpée plus précisément.
Le meilleur wedge de départ n’est pas encore totalement verrouillé, même si les documents convergent vers :
- micro-SaaS B2B simple à moyen ;
- mono-workflow ou workflow standard ;
- projet déjà réel ;
- avant-code fort + pendant-build léger.

## JTBD audités
### JTBD central retenu
Quand mon prototype IA commence à devenir un vrai produit, je veux une structure claire et une continuité fiable afin de continuer sans casser l’existant ni repartir de zéro.

### JTBD principaux cohérents
- transformer un prototype fragile en base SaaS structurée ;
- sortir d’un builder rapide sans perdre l’élan ;
- reprendre le projet sans se reperdre ;
- transmettre sans rebuild ;
- modifier sans provoquer de régressions partout.

### Lecture d’audit
Les JTBD sont globalement cohérents avec :
- la proposition de valeur ;
- le MVP ;
- les flows ;
- la logique business.

Ce ne sont pas des JTBD décoratifs.
Ils décrivent bien un gain concret :
- contrôle ;
- reprise ;
- transmission ;
- réduction du risque de casse.

## Proposition de valeur auditée
### Promesse centrale observée
Transformer un proto IA fragile en base SaaS claire, exploitable et transmissible sans rebuild aveugle.

### Ce qui rend la promesse crédible
La promesse V1 est raisonnablement contenue :
- ordre avant la casse totale ;
- continuité légère pendant le build ;
- pas de promesse “on gère tout le build” ;
- pas de promesse de génération magique.

### Point fort majeur
Le projet n’essaie plus de gagner sur le terrain saturé de la vitesse pure.
Il essaie de gagner sur :
- la cohérence ;
- la mémoire de décision ;
- la source de vérité ;
- la transmission.

C’est cohérent avec le marché décrit comme bruyant sur “idea to app”, mais plus mal servi sur “idea to reliable project”.

## Modèle économique audité
## État actuel des documents
Il y a eu une évolution réelle du modèle économique.

### Historique
Les documents amont et les premières contraintes décrivaient un système de crédits.

### État recommandé plus récent
Les documents business les plus récents déplacent le modèle vers :
- projet actif comme unité de valeur ;
- revenu logiciel récurrent ;
- continuité ;
- logique potentiellement hybride ;
- crédits archivés comme hypothèse historique et non comme modèle cible.

### Lecture d’audit
Ce déplacement est stratégiquement cohérent, car :
- le payeur est le même que l’utilisateur en V1 ;
- le projet crée de la valeur sur un projet déjà réel ;
- l’unité “projet actif” est plus concrète que “workspace” ou “crédits” ;
- la façade vendable est le dossier canonique exportable ;
- la profondeur économique à moyen terme vient de la continuité.

## Cohérences observées
### 1. Cohérence problème -> cible
Oui.
La cible retenue souffre bien du problème décrit.
Le moment de douleur est crédible :
- post-proto ;
- quand la complexité monte ;
- quand il faut reprendre, migrer, transmettre ou préparer la suite.

### 2. Cohérence cible -> JTBD
Oui.
Les JTBD décrivent bien les tâches critiques de cette cible :
- remettre au carré ;
- reprendre ;
- modifier sans casser ;
- transmettre ;
- sortir d’un builder rapide.

### 3. Cohérence JTBD -> promesse
Oui.
La promesse de “base de vérité exploitable + continuité légère” répond bien à ces JTBD.

### 4. Cohérence promesse -> scope V1
Plutôt oui.
Le scope V1 inclus et hors scope est de plus en plus discipliné :
- builders déjà initiés ;
- SaaS simples à moyens ;
- avant-code fort ;
- continuité légère ;
- hors scope enterprise, ultra-spécifique, build autonome fort.

### 5. Cohérence promesse -> modèle économique
Partiellement oui.
Le modèle économique recommandé colle à la valeur vendue :
- dossier canonique exportable ;
- projet actif ;
- continuité ;
- extensions par collaboration / handoff / multi-projets.

Mais le pricing final reste dépendant de validations terrain.

## Fragilités observées
### 1. Tension historique crédits -> projet actif
Le projet a bien évolué d’une logique crédits vers une logique projet actif / revenu récurrent.
Cette transition est compréhensible et probablement saine, mais elle doit être traitée explicitement comme un changement de doctrine.
Sinon le projet peut raconter deux histoires économiques incompatibles.

### 2. Tension “livrable ponctuel” vs “continuité récurrente”
Les documents convergent vers :
- façade vendable = dossier exportable ;
- profondeur économique = continuité.
C’est cohérent, mais pas totalement verrouillé.
Le risque est de vendre un one-shot tout en essayant de monétiser un récurrent non encore prouvé.

### 3. Tension fréquence du besoin -> abonnement mensuel
Les unit economics disent clairement :
- usage intense par pics ;
- fréquence non uniforme ;
- support potentiellement plus élevé qu’un SaaS banal ;
- danger d’un flat trop généreux.
Donc un abonnement mensuel pur n’est pas absurde, mais il n’est pas encore suffisamment prouvé pour être traité comme certitude.

### 4. Tension profondeur documentaire -> adoption
La profondeur documentaire est un point de force stratégique.
Mais elle peut aussi créer :
- surcharge perçue ;
- friction d’onboarding ;
- objection “je ne veux pas plus de doc”.

### 5. Tension wedge narratif -> wedge acheteur réel
Le segment post-Lovable / post-builder bloqué est prometteur.
Mais plusieurs documents disent explicitement qu’il faut encore vérifier s’il s’agit d’un vrai segment acheteur ou d’un angle séduisant.

## Niveau global de robustesse
### Ce qui est robuste
- le noyau problème / cible / JTBD ;
- la proposition de valeur ;
- la différenciation comme couche de fiabilité plutôt que builder de plus ;
- la direction générale du modèle économique ;
- la discipline progressive du scope V1.

### Ce qui reste fragile
- la phrase de douleur la plus achetable ;
- le wedge exact de départ ;
- l’objet vendu le plus vite compris et le plus vite acheté ;
- le moment précis du paiement ;
- le packaging économique final ;
- le corridor de prix acceptable ;
- l’estimation réelle des coûts variables et du support.

## Conclusion d’audit
Le projet est **stratégiquement cohérent dans son noyau**.

Il ne présente pas un chaos de fondations.
En revanche, il présente encore plusieurs points sensibles concentrés sur :
- la formulation d’offre ;
- le packaging économique ;
- le degré exact de profondeur documentaire ;
- le point d’entrée produit et commercial ;
- la validation de la fréquence d’usage et de la willingness to pay.

## Niveau global de robustesse
**Robustesse globale : moyenne à bonne.**

### Pourquoi
- le problème, la cible et la valeur sont bien alignés ;
- la différenciation est intelligible ;
- le modèle business a gagné en maturité ;
- mais le pricing final et la forme économique exacte restent encore dépendants de validations terrain.

## Statut recommandé
**Transmission autorisée : Oui sous hypothèses.**
