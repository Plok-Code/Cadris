# Registre de certitude

## Confirmé
- Le projet ne doit pas être positionné comme un simple app builder pour débutants.
- La valeur centrale n’est pas de remplacer Codex ou les agents de code, mais de mieux les alimenter et les guider.
- La cible connaît déjà un peu les outils IA de build.
- Le terrain de jeu retenu est le **SaaS complet avec vraie architecture long terme**.
- Le point de friction le plus crédible apparaît quand le projet cesse d’être un simple proto et commence à devenir complexe.
- Le segment “novice total” est un mauvais point d’entrée.
- Le produit semble plus crédible comme couche de structuration, de documentation exploitable et d’ordre des décisions.

## Hypothèses de travail
- Le meilleur ICP de départ est le **vibecodeur débutant à intermédiaire qui a déjà fait une petite app et veut maintenant construire un vrai SaaS**.
- Un sous-segment particulièrement prometteur est l’utilisateur passé par Lovable ou un builder IA équivalent puis frustré par le plafond rencontré.
- La douleur d’achat principale est moins “comment démarrer ?” que “comment éviter que tout parte en vrille quand le projet grossit ?”.
- Une promesse du type “passer du proto bricolé au produit sérieux” pourrait mieux performer qu’une promesse centrée sur la simple vitesse.
- Impact : ces hypothèses orientent fortement la segmentation, le wedge et le message de la V1.
- Pourquoi ces hypothèses ont été retenues :
  - elles sont cohérentes avec le cadrage utilisateur donné en conversation ;
  - elles sont cohérentes avec les signaux marché observés sur l’écart entre prototypage rapide et montée en complexité ;
  - elles différencient mieux le produit des builders rapides grand public.

## Inconnus
- La douleur d’achat dominante exacte :
  - architecture long terme ;
  - dossier canonique / documentation ;
  - ordre des décisions ;
  - réduction du risque de rebuild ;
  - continuité entre cadrage et exécution.
- Le poids réel du sous-segment “post-Lovable plateau”.
- Le premier type de SaaS le plus naturel à cadrer dans la V1.
- Le moment exact où l’utilisateur est prêt à payer :
  - avant de coder sérieusement ;
  - après les premiers signaux de chaos ;
  - quand une première version existe déjà mais devient difficile à maintenir.
- Pourquoi ces points restent inconnus :
  - pas encore de validation terrain directe suffisante ;
  - signaux marché utiles mais non quantifiés au niveau nécessaire.
- Quel impact potentiel :
  - mauvais message ;
  - cible trop large ;
  - promesse trop abstraite ;
  - onboarding mal séquencé.

## Bloquants
- Définir la douleur d’achat principale en une phrase simple.
- Définir le premier wedge SaaS explicitement inclus pour la V1.
- Définir jusqu’où le produit agit :
  - avant le code seulement ;
  - ou aussi pendant l’exécution et les itérations.
- Pourquoi c’est bloquant :
  - sans cela, la priorisation de cible reste correcte mais le positionnement de la V1 restera flou ;
  - le GPT suivant risque de bâtir un messaging trop large.
- Ce qu’il faut obtenir pour débloquer :
  - 5 à 10 entretiens ou signaux utilisateur qualifiés ;
  - une hiérarchie claire des douleurs ;
  - 3 cas inclus et 3 cas exclus pour le premier wedge.

## Statut de transmission
- Transmission autorisée : **Oui sous hypothèses**
- Raison :
  - la segmentation est suffisamment structurée pour avancer ;
  - mais elle repose encore sur un noyau d’hypothèses de marché à confirmer rapidement.

## Handoff au GPT suivant
Le GPT 04 doit traiter comme hypothèse principale :
**la meilleure cible V1 n’est pas le novice total, mais le builder déjà initié aux outils IA qui veut passer d’un proto bricolé à un vrai SaaS sans perdre le contrôle de la complexité.**
