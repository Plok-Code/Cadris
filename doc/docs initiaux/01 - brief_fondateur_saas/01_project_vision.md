# Vision du projet

## Statut du nom
**Inconnu à ce stade.**  
Le nom du projet n'est pas encore défini. Ce point n'empêche pas le cadrage fondateur.

## Résumé du projet
Le projet est un système multi-agents spécialisé, conçu pour transformer une idée de projet numérique en un corpus documentaire très poussé, puis guider son exécution par LLM de manière itérative et plus fiable.

## Point de départ retenu
Le point d'entrée retenu est le **SaaS**, traité avec une ambition de couverture maximale.  
La V2 visera d'autres types de projets numériques, mais la V1/prototype doit être capable de cadrer un SaaS de manière très profonde.

## Pourquoi ce projet doit exister
L'engouement autour des IA et du vibe coding permet à de nombreux débutants de lancer des projets numériques.  
Mais, laissés à eux-mêmes, les LLM ont tendance à :
- choisir une stack sans cadre suffisamment solide ;
- improviser le déroulement du projet ;
- produire des sorties incohérentes ou trop fragiles pour des projets sérieux ;
- donner une illusion de progression alors que les fondations restent faibles.

Le projet doit donc exister pour transformer un usage opportuniste des IA en une démarche plus robuste, structurée et professionnelle.

## Cible principale
Les **vibecodeurs débutants dans l'entrepreneuriat ou la création de projets numériques**, en particulier ceux qui veulent lancer un SaaS sans maîtriser eux-mêmes tout le cadrage stratégique, produit et technique.

## Transformation promise
Le projet promet de faire passer l'utilisateur :
- d'une idée encore floue ou partiellement structurée ;
- à un **dossier d'exécution très poussé**, cohérent, exploitable par des LLM ;
- puis à une **exécution guidée**, où un chatbot peut suggérer le prochain prompt, demander des vérifications et aider à garder le cap.

## Ce que le projet n'est pas
Le projet n'est pas :
- un simple générateur de documents génériques ;
- un chatbot qui improvise un produit au fil de l'eau ;
- un outil de build autonome sans cadrage préalable ;
- un simple assistant de prompts déconnecté du besoin utilisateur réel.

## Vision de la valeur créée
La valeur principale n'est pas seulement de produire "beaucoup de documents", mais de :
- réduire la dérive des choix techniques et produit ;
- augmenter la cohérence entre besoin utilisateur, cadrage, stack et exécution ;
- rendre un projet plus sérieux, plus transmissible et plus exécutable par des agents ou des équipes.

## Vision du parcours utilisateur
Le parcours cible, à ce stade, ressemble à ceci :
1. l'utilisateur arrive avec une idée de SaaS ;
2. le système pose une série de questions approfondies ;
3. plusieurs agents spécialisés traitent chaque couche du problème ;
4. le système produit un corpus documentaire complet ;
5. l'utilisateur utilise ce corpus avec un LLM de build ;
6. un chatbot d'aval peut ensuite analyser les sorties et suggérer le prochain prompt ou les vérifications nécessaires.

## Vision de monétisation à ce stade
Le modèle économique retenu à ce stade est un **système de crédits** :
- 10 crédits gratuits au départ ;
- consommation visible en temps réel ;
- alerte claire lorsqu'il reste 2 crédits ;
- à 0 crédit, l'entretien se met en pause ou passe en mode brouillon non livrable ;
- pour obtenir le dossier final exportable, l'utilisateur doit recharger.

## Portée documentaire cible
Pour le périmètre SaaS, l'ambition est de couvrir un corpus documentaire très large, incluant au minimum :
- stratégie ;
- cadrage produit ;
- UX/UI ;
- technique ;
- data/IA si nécessaire ;
- juridique et conformité ;
- exécution ;
- go-to-market ;
- dossier d'exécution consolidé ;
- guidance de prompts et contrôle qualité pour l'aval.

Des documents additionnels peuvent être générés selon le cas, par exemple :
- plan de migration ;
- stratégie de versioning ;
- matrice de permissions ;
- SLA/SLO ;
- plan de tests ;
- plan d'observabilité ;
- plan de support ;
- plan de reprise après incident ;
- catalogue d'événements analytics ;
- documentation de change management ;
- documentation d'onboarding interne.

## Statut de fiabilité de cette vision
### Confirmé
- Le projet cible d'abord le SaaS.
- Le projet veut produire un corpus documentaire très poussé.
- Le projet inclut une boucle d'exécution guidée après la production des documents.
- Le projet vise des vibecodeurs débutants en entrepreneuriat.

### Hypothèses de travail
- Le système pourra couvrir "le SaaS à 100%" sans devoir segmenter fortement par sous-type de SaaS.
- La profondeur documentaire sera perçue comme une force et non comme une surcharge.

### Inconnu
- Le nom final du projet.
- Le niveau de standardisation exact entre SaaS simples et SaaS complexes.
- Les limites précises du premier périmètre opérationnel.

### Bloquant
- Aucun bloquant majeur pour transmettre la vision.
- En revanche, le GPT suivant devra rapidement vérifier la faisabilité du "SaaS à 100%" en première itération.
