# Handoff au GPT 02

## Résumé exécutif
Le projet vise à construire un système multi-agents capable de transformer une idée de projet numérique en documentation très poussée, puis d'accompagner son exécution assistée par LLM.  
Le point de départ retenu est le **SaaS**, avec une ambition de couverture maximale.  
La cible principale est constituée de vibecodeurs débutants dans l'entrepreneuriat ou la création de projets numériques, qui veulent construire vite avec l'IA mais sans subir les dérives habituelles : mauvaise stack, projet mal cadré, documents faibles ou contradictoires.

Le produit ne doit pas seulement générer des documents. Il doit :
- cadrer profondément le projet ;
- produire un dossier d'exécution consolidé ;
- permettre à un LLM de build d'agir avec beaucoup moins d'improvisation ;
- proposer ensuite une boucle d'aval où un chatbot peut suggérer le prochain prompt et contrôler la qualité des sorties.

## Problème principal
Les LLM, laissés trop libres dans un contexte de vibe coding, choisissent eux-mêmes la stack, l'ordre des décisions et parfois la logique produit.  
Cela rend les projets fragiles, surtout dès qu'ils deviennent sérieux, complexes ou durables.

## Cible principale
Vibecodeurs débutants en entrepreneuriat ou en création de projets numériques, avec un intérêt fort pour les IA de build, mais sans méthode de cadrage suffisamment robuste.

## Promesse centrale
Transformer une idée de projet en un ensemble de documentation très poussée pour les LLM, afin de rendre l'exécution plus professionnelle, plus cohérente et moins fragile.

## Contraintes initiales
- Stade : prototype
- Périmètre initial : SaaS
- Ambition documentaire : maximale
- Niveau technique du porteur : avancé
- Ressources humaines disponibles : front, back, ML, data, infra, devops
- Budget/temps : pas de contrainte forte exprimée
- Contrainte réelle : périmètre, qualité, cohérence et industrialisation

## Points confirmés
- Le nom n'est pas encore défini.
- Le projet commence par le SaaS avant extension à d'autres types de projets.
- Le système doit produire le maximum de documents pertinents pour un SaaS, avec génération additionnelle selon les cas.
- Le livrable le plus précieux est un dossier d'exécution consolidé.
- Le chatbot d'aval doit suggérer le prochain prompt et aussi contrôler la qualité de la sortie.
- Les critères de succès centraux sont : qualité perçue très élevée, stack pertinente, dossier cohérent.
- Les signaux d'échec centraux sont : documents contradictoires, faible qualité documentaire, mauvaise stack.
- Le modèle économique retenu en première intention est un système de crédits lisible et non piégeux.

## Hypothèses de travail
- Le marché ressent suffisamment la douleur pour adopter un tel produit.
- Le SaaS peut être traité de façon très complète dans une première version avant spécialisation plus fine.
- La profondeur documentaire constitue une valeur perçue et non une surcharge.
- Le modèle de crédits restera compatible avec une relation de confiance forte.

## Inconnus
- Nom final du projet
- Métriques qualité opérationnelles
- Règles précises de pertinence de stack
- Degré de segmentation interne du périmètre SaaS
- Structure finale normalisée du dossier exportable
- Prix et packs de crédits

## Bloquants
- Définir une grille de qualité observable pour juger les dossiers
- Définir les critères de pertinence de stack
- Cadrer le vrai périmètre du "SaaS à 100%"
- Définir la structure canonique du dossier exportable
- Clarifier la doctrine de fonctionnement du chatbot d'aval
- Tester la viabilité du modèle crédits

## Niveau de fiabilité du brief
**Moyen à bon.**

### Pourquoi
Le noyau stratégique est clair :
- cible ;
- problème ;
- promesse ;
- périmètre initial ;
- logique de valeur ;
- logique générale de monétisation.

Mais plusieurs éléments critiques restent encore au niveau de l'intention :
- critères qualité ;
- cadre exact du périmètre ;
- règles de recommandation technique ;
- économie détaillée ;
- normalisation finale des livrables.

## Ce que le GPT 02 doit absolument vérifier en priorité
1. Transformer les critères de succès qualitatifs en critères observables.
2. Définir comment le système évalue et justifie une stack pertinente.
3. Encadrer le périmètre "SaaS à 100%" sans trahir l'ambition du porteur.
4. Structurer le dossier final exportable en une hiérarchie stable et exploitable.
5. Définir les comportements du chatbot d'aval : suggestion, contrôle, alerte, insistance.
6. Tester la cohérence du modèle crédits avec la valeur produite et la confiance utilisateur.

## Note de transmission
Transmission autorisée au GPT 02, **mais sous hypothèses**.  
Le projet est suffisamment cadré pour passer à une phase suivante de formalisation plus détaillée, à condition de ne pas maquiller les inconnus en certitudes.
