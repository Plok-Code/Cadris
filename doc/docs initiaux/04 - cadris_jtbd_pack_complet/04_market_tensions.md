# Tensions de marché

## Lecture d’ensemble
Le marché paraît à la fois :
- très dynamique ;
- très encombré sur certains usages ;
- encore imparfaitement résolu sur la fiabilité globale.

Autrement dit :
il y a probablement une opportunité, mais pas dans un espace calme.
Il faut entrer dans une zone tendue, déjà active, et choisir un angle très net.

## Tension 1 — Explosion de l’offre, confiance incomplète
D’un côté, l’adoption des outils IA dans le développement continue de monter.
De l’autre, la confiance dans les sorties et la capacité à traiter des tâches complexes restent limitées.

### Ce que cela crée
- un marché ouvert ;
- mais aussi des utilisateurs plus prudents qu’on pourrait le croire.

### Effet sur le projet
Bonne nouvelle :
- la douleur n’est pas théorique.

Mauvaise nouvelle :
- le produit devra prouver une vraie amélioration de qualité, pas seulement une promesse.

## Tension 2 — Le marché récompense la vitesse, alors que le besoin réel peut être la cohérence
La plupart des leaders visibles vendent :
- un prototype en minutes ;
- une app en quelques prompts ;
- une accélération spectaculaire.

Le projet, lui, vendrait davantage :
- du cadrage ;
- de la mémoire de décision ;
- de la cohérence documentaire ;
- du contrôle de dérive.

### Ce que cela crée
Une tension de marketing importante :
- ce que le marché applaudit n’est pas toujours ce qui manque le plus.

## Tension 3 — Saturation sur “idea to app”, fragmentation sur “idea to reliable project”
Le segment “générer une app ou un prototype avec IA” semble déjà encombré.
En revanche, la chaîne complète qui va jusqu’au dossier canonique et au maintien du cap paraît plus fragmentée.

### Effet sur le projet
- Entrer comme app builder générique serait dangereux.
- Entrer comme couche de fiabilité peut être plus défendable.

## Tension 4 — Débutants demandeurs, mais difficiles à servir
Les débutants ressentent plausiblement le plus la douleur.
Mais ils sont aussi :
- les plus sensibles à la complexité ;
- les moins capables d’évaluer la qualité profonde ;
- les plus attirés par les promesses de simplicité immédiate.

### Effet sur le projet
Le produit doit être assez robuste pour sécuriser les décisions, sans devenir lui-même un tunnel documentaire intimidant.

## Tension 5 — Plus la valeur est sérieuse, plus le périmètre devient difficile
Plus le projet cherche à couvrir un SaaS “très sérieusement”, plus il doit traiter :
- architecture ;
- UX ;
- priorisation ;
- conformité ;
- data ;
- exploitation ;
- go-to-market ;
- handoff agents/humains.

### Effet sur le projet
La profondeur crée la valeur.
Mais la profondeur crée aussi le risque de lourdeur, de lenteur et d’industrialisation difficile.

## Saturation ou fragmentation supposée
## Saturation probable
- build d’apps par prompt ;
- prototypage rapide ;
- génération de code assistée ;
- outils généralistes de doc et de coordination.

## Fragmentation probable
- transition entre idée et spécification robuste ;
- mémoire de décision transférable ;
- contrôle de cohérence entre intention, documentation et sorties des agents ;
- méthode prête à l’emploi pour débutants ambitieux.

## Pain points potentiellement non couverts
- savoir quel document fait foi ;
- garder une cohérence de bout en bout ;
- éviter qu’un agent dérive sans signal fort ;
- transmettre le projet sans tout réexpliquer ;
- justifier les choix techniques et produit de manière intelligible.

## Signaux favorables
- la demande pour des outils de build IA progresse rapidement ;
- les acteurs majeurs poussent déjà des modes planification, review, règles et tests ;
- cela valide que la couche de contrôle et de cadrage devient un vrai sujet, pas un détail ;
- la confiance et la qualité restent suffisamment imparfaites pour laisser une fenêtre à une offre plus structurante.

## Signaux défavorables
- les concurrents peuvent étendre vite leurs fonctionnalités vers le cadrage ;
- le marché peut juger qu’un bon workflow dans des outils existants suffit ;
- la promesse documentaire peut sembler lourde ;
- le terme “vibecodeur débutant” peut être réel comme segment d’usage, mais trop faible comme segment d’achat.

## Lecture honnête
L’opportunité initiale paraît plausible, mais elle n’est pas encore démontrée.

### Ce qui semble le plus solide
- la douleur de cohérence et de dérive existe probablement ;
- le marché n’a pas fini de croître ;
- l’existant couvre mal la chaîne complète de fiabilité.

### Ce qui reste fragile
- la capacité à faire payer pour cette couche ;
- le bon segment d’entrée ;
- le bon niveau de profondeur avant surcharge ;
- la vitesse à laquelle les leaders du build ajouteront des briques concurrentes.

## Confirmé
- Le marché n’est ni vierge ni mature au point d’être figé.
- Il y a déjà beaucoup d’énergie sur la création d’apps avec IA.
- Le projet devra donc éviter un positionnement générique.

## Hypothèses de travail
- La tension confiance/qualité deviendra un axe de différenciation important.
  - Impact : fort.
  - Pourquoi cette hypothèse a été retenue : le marché adopte vite, mais ne fait pas encore totalement confiance aux sorties.

- La fragmentation entre planification, documentation et build restera assez forte pour laisser une fenêtre à un acteur spécialisé.
  - Impact : très fort.
  - Pourquoi cette hypothèse a été retenue : l’existant semble encore composé de briques plus que d’une chaîne unifiée.

## Inconnus
- Vitesse de convergence des leaders vers une offre plus complète.
- Sensibilité réelle du marché à la promesse de dossier canonique.
- Meilleur wedge d’entrée : segment, usage, complexité, moment du parcours.

## Bloquants
- Le principal bloquant reste commercial :
  - peut-on vendre une couche de fiabilité distincte, ou le marché attend-il que cette couche soit incluse dans les builders existants ?
