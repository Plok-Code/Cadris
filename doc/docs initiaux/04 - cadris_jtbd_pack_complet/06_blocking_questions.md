# Questions bloquantes restantes

## 1. Quelle phrase décrit le mieux la douleur d’achat principale ?
### Pourquoi cela bloque
Le projet peut être acheté pour plusieurs raisons proches mais différentes :
- “je ne veux pas rebuild plus tard” ;
- “je veux une vraie architecture long terme” ;
- “je veux enfin une source de vérité claire” ;
- “je veux arrêter de me perdre quand le projet grossit”.

Sans hiérarchie claire, le message restera flou.

### Ce qu’il faut obtenir
- une douleur principale ;
- une formulation utilisateur simple ;
- 1 ou 2 douleurs secondaires maximum.

---

## 2. Quel wedge SaaS exact sert-on en premier ?
### Pourquoi cela bloque
“SaaS complet avec vraie architecture long terme” reste encore trop large.

### Ce qu’il faut obtenir
- 3 à 5 types de SaaS explicitement inclus ;
- 3 à 5 types de SaaS explicitement exclus ;
- un niveau de complexité acceptable pour la V1.

---

## 3. Le produit intervient-il seulement avant le code, ou aussi pendant l’exécution ?
### Pourquoi cela bloque
Le produit peut être perçu soit comme :
- un système de cadrage et de documentation ;
- soit comme un système de cadrage + pilotage continu du build.

Ce choix change fortement l’ICP, l’onboarding et la proposition de valeur.

### Ce qu’il faut obtenir
- une limite claire de périmètre ;
- ce que le produit produit concrètement au moment 1 ;
- ce qu’il ne prétend pas encore faire.

---

## 4. Le sous-segment “post-Lovable / post-builder bloqué” est-il un vrai wedge de marché ?
### Pourquoi cela bloque
Ce sous-segment paraît très intéressant, mais il ne faut pas le transformer trop vite en certitude absolue.

### Ce qu’il faut obtenir
- témoignages directs ou entretiens ;
- motifs de blocage récurrents ;
- vocabulaire exact utilisé par ces utilisateurs.

---

## 5. Quel niveau de sophistication documentaire est réellement perçu comme utile ?
### Pourquoi cela bloque
Trop de profondeur documentaire peut faire peur ; trop peu peut sembler insuffisant pour un vrai SaaS.

### Ce qu’il faut obtenir
- le niveau minimum de livrable perçu comme “suffisamment puissant” ;
- le format jugé le plus actionnable pour travailler ensuite dans Codex / IDE / repo.

---

## Priorité de résolution
1. douleur principale ;
2. wedge SaaS initial ;
3. périmètre avant-code vs pendant exécution ;
4. validation du sous-segment post-Lovable ;
5. niveau de profondeur documentaire.
