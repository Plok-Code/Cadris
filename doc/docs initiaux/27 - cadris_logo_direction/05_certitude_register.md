# 05_certitude_register

# Registre de certitude

## Confirme
- Cadris a besoin d'un vrai logo avec symbole, pas d'un logotype seul.
- Le logo doit rester sobre, structurel, lisible et anti-hype.
- Le symbole ne doit pas raconter l'IA magique, la vitesse ou la complexite technique.
- Les usages critiques sont au minimum : header, favicon, avatar et documents.
- Le logo doit vivre dans un systeme visuel deja etabli : palette minerale, accent petrol, langage de cadres et de reperes, ton expert accessible.
- Le logo doit rester compatible avec une forte reduction.
- La base typographique de travail pour le logotype est `Public Sans + IBM Plex Mono`.

## Hypotheses de travail
- H1 - L'usage critique principal est `site + header + favicon`.
- Impact : favorise un symbole simple, une signature horizontale et une reduction forte.
- Pourquoi cette hypothese a ete retenue : l'utilisateur n'a pas tranche un support unique, mais ces usages sont les plus probables au lancement.

- H2 - La meilleure architecture logo est une `combination mark sobre`.
- Impact : impose de construire a la fois un signe autonome et un logotype stable.
- Pourquoi cette hypothese a ete retenue : l'utilisateur a explicitement demande un vrai logo et non un simple logotype.

- H3 - Les trois familles doivent etre testees en parallele avant choix final.
- Impact : le dossier 27 sert a cadrer une exploration comparative, pas a figer prematurement une seule solution.
- Pourquoi cette hypothese a ete retenue : l'utilisateur a demande plusieurs sous-idees sur les trois pistes.

- H4 - La famille `monogramme structure` est la plus prometteuse a ce stade.
- Impact : elle sert de recommandation initiale sans exclure les familles cadre / repere et couche / assemblage.
- Pourquoi cette hypothese a ete retenue : elle combine mieux memorisation, reduction et possibilite de vrai signe.

## Inconnus
- I1 - Le support qui deviendra vraiment dominant au lancement n'est pas confirme.
- Pourquoi ce point reste inconnu : aucun usage unique n'a ete tranche.
- Quel impact potentiel : peut changer le niveau de priorite entre header, favicon et document.

- I2 - Le besoin exact d'une app icon produit reste non confirme.
- Pourquoi ce point reste inconnu : le projet parle de produit numerique, mais pas encore de package mobile ou desktop concret.
- Quel impact potentiel : change la pression sur le symbole autonome.

- I3 - Le niveau de simplification acceptable pour le symbole n'est pas encore teste.
- Pourquoi ce point reste inconnu : aucune execution graphique n'a encore ete comparee en reduction.
- Quel impact potentiel : certaines familles plausibles sur le papier peuvent s'effondrer a 16px.

## Bloquants
- B1 - Aucune famille de symbole n'a encore ete testee graphiquement en petite taille.
- Pourquoi c'est bloquant : impossible de valider une direction logo sans test de reduction.
- Ce qu'il faut obtenir pour debloquer : produire et comparer les trois familles avec leurs sous-pistes.

- B2 - Le besoin reel d'une app icon autonome n'est pas tranche.
- Pourquoi c'est bloquant : peut influer sur la simplification du symbole.
- Ce qu'il faut obtenir pour debloquer : decision sur les surfaces de lancement prioritaires.

## Statut de transmission
- Transmission autorisee : Oui sous hypotheses
- Raison : la direction logo est suffisamment cadre pour lancer une exploration graphique, mais pas encore pour valider une solution unique sans tests.
