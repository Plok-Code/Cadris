# 07_handoff_to_gpt_28

## Resume executif

La direction logo de Cadris est maintenant suffisamment cadree pour lancer une exploration graphique serieuse.

Le point cle n'est pas de dessiner un logo "style".
Le point cle est de produire un vrai logo compatible avec les usages reels du projet, tout en restant coherent avec :
- une marque editoriale et structurelle ;
- un ton expert accessible ;
- un systeme visuel sobre, mineral et produit-compatible ;
- une promesse centrale de `cadre clair`.

## Direction logo retenue

Direction de fond :
**combination mark sobre**

Concretement :
- un logotype principal stable ;
- un symbole autonome ;
- une relation forte entre les deux ;
- un niveau de symbolique modere-faible.

## Contraintes principales

- lisibilite forte en header ;
- reduction severe en favicon et avatar ;
- fonctionnement en monochrome ;
- zero imaginaire IA generique ;
- zero effet corporate froid ;
- zero complexite gratuite.

## Variantes necessaires

- signature principale horizontale ;
- logotype seul ;
- symbole seul ;
- monochrome ;
- eventuelle variante compacte si justifiee.

## Regles favicon / app icon

- le favicon doit etre base sur le symbole seul ;
- il doit exister en micro-version si necessaire ;
- la reduction `16px` est critique ;
- l'app icon n'est pas encore confirmee comme surface prioritaire, donc ne doit pas sur-determiner tout le systeme.

## Familles de symbole a tester

### Famille A - Cadre / repere
Sous-pistes :
- A1 Cadre ouvert
- A2 Double repere d'angle
- A3 Cadre + noyau
- A4 Repere de marge

### Famille B - Couche / assemblage
Sous-pistes :
- B1 Cartes empilees
- B2 Modules relies
- B3 Superposition verrouillee
- B4 Pile + repere

### Famille C - Monogramme structure
Sous-pistes :
- C1 C cadre
- C2 C + repere
- C3 CA structure
- C4 C en couches

## Recommandation initiale

Ordre de priorite pour exploration :
1. Monogramme structure
2. Cadre / repere
3. Couche / assemblage

Ce n'est pas un verdict final.
C'est une recommendation de production pour obtenir vite des pistes testables.

## Points confirmes

- Cadris doit avoir un vrai logo avec signe ;
- le logo ne doit pas raconter tout le produit ;
- le symbole doit rester simple et reductible ;
- le logo doit d'abord reussir les usages web et petite taille ;
- la marque doit rester structurelle, claire, fiable et non theatrale.
- la base typographique de travail du logotype est `Public Sans + IBM Plex Mono`.

## Hypotheses de travail

- usage critique principal : header + favicon + avatar ;
- combination mark comme meilleure architecture ;
- tests en parallele des trois familles ;
- monogramme structure comme piste la plus prometteuse a priori.

## Inconnus

- famille symbole la plus performante en reduction reelle ;
- importance reelle de l'app icon a court terme ;
- niveau de simplification micro-usage acceptable ;
- regle definitive entre version monochrome et variante petrol du logo.

## Bloquants

- tester graphiquement les trois familles ;
- clarifier la priorite reelle de l'app icon.

## Niveau de fiabilite

- Niveau de fiabilite : Bon
- Raison : le cadrage s'appuie sur l'ensemble du systeme visuel, sur le messaging et sur les contraintes d'usage, mais il reste volontairement exploratoire tant que les tests de reduction n'ont pas eu lieu.

## Ce que le GPT 28 doit traduire en principes UI

1. Prevoir un header ou la signature horizontale reste legere et lisible.
2. Prevoir des emplacements ou le symbole seul peut vivre proprement : favicon, avatar, repere de navigation.
3. Eviter de construire l'UI autour d'un logo envahissant ou narratif.
4. Maintenir la priorite de la clarte systeme sur la demonstration de marque.
5. Prevoir que certaines vues denses utiliseront plutot le symbole seul que le logotype complet.
6. Garder une coherence stricte entre formes UI, iconographie et logique du symbole retenu.

## Ce que le GPT 28 doit traiter avec prudence

- toute UI qui dependrait trop d'un symbole encore non arbitre ;
- toute mise en page heroique qui surestime le logo ;
- toute traduction UI qui ferait glisser Cadris vers un app builder flashy, un cockpit technique ou un cabinet froid.
