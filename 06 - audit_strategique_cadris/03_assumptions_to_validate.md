# Hypothèses à valider

## Objet
Ce document liste les hypothèses critiques qui influencent directement :
- la capacité à vendre ;
- la clarté de l’offre ;
- la structure économique ;
- la rétention ;
- la marge.

## Hypothèse 1 — La meilleure douleur d’achat est la réduction du risque de rebuild / dérive
### Pourquoi c’est sensible
Les documents convergent vers plusieurs formulations proches :
- éviter le rebuild ;
- arrêter la dérive ;
- retrouver une source de vérité ;
- reprendre ou transmettre sans se reperdre.

La hiérarchie exacte n’est pas encore prouvée.

### Impact si c’est faux
- mauvais messaging ;
- mauvaise landing ;
- offre perçue comme abstraite ;
- difficulté à justifier le prix.

### Ce qu’il faut tester
- réactions à plusieurs formulations ;
- préférence spontanée des prospects ;
- langage exact utilisé par la cible ;
- capacité de chaque formulation à déclencher un rendez-vous, un waitlist join ou un paiement.

## Hypothèse 2 — Le builder post-proto est plus acheteur que l’utilisateur en idéation pure
### Pourquoi c’est sensible
Toute la logique produit récente suppose que le meilleur moment est :
- projet déjà réel ;
- proto déjà sorti ;
- complexité qui monte.

### Impact si c’est faux
- mauvais point d’entrée ;
- mauvaise acquisition ;
- mauvaise structure d’onboarding ;
- offre trop tardive ou trop précoce.

### Ce qu’il faut tester
- entretiens segmentés par stade ;
- tests de message par contexte d’entrée ;
- différence d’intérêt entre “idée”, “proto” et “pré-handoff”.

## Hypothèse 3 — Le bon wedge initial est un micro-SaaS B2B simple à moyen
### Pourquoi c’est sensible
Le projet ne peut pas raisonnablement ouvrir “tout SaaS” au départ.
Il faut vérifier que ce wedge est :
- assez fréquent ;
- assez douloureux ;
- assez solvable ;
- assez industrialisable.

### Impact si c’est faux
- scope V1 mal calibré ;
- acquisition floue ;
- cas trop hétérogènes ;
- produit trop large trop tôt.

### Ce qu’il faut tester
- 3 cas inclus ;
- 3 cas exclus ;
- signaux de préférence sur un angle simple ;
- compréhension immédiate du périmètre.

## Hypothèse 4 — Le dossier canonique exportable est le meilleur objet de vente initial
### Pourquoi c’est sensible
Les documents le traitent comme la façade vendable la plus concrète.
Mais cela reste à vérifier face à :
- la mémoire continue ;
- la continuité de décision ;
- le maintien de cohérence.

### Impact si c’est faux
- packaging de valeur mal choisi ;
- mauvaise promesse d’entrée ;
- confusion entre livrable et usage continu.

### Ce qu’il faut tester
- ce qui est compris le plus vite ;
- ce qui semble le plus actionnable ;
- ce qui paraît le plus “achetable” sans démonstration longue.

## Hypothèse 5 — Une continuité légère suffit à créer de la rétention
### Pourquoi c’est sensible
Le modèle économique le plus séduisant repose sur une couche récurrente.
Mais le besoin peut rester trop ponctuel pour certains comptes.

### Impact si c’est faux
- rétention faible ;
- mensuel difficile à tenir ;
- dépendance excessive au one-shot.

### Ce qu’il faut tester
- fréquence de revisite ;
- cas de retour naturels ;
- valeur perçue de la mémoire / versioning / guidance légère ;
- ratio usage initial / usage continu.

## Hypothèse 6 — Projet actif est la meilleure unité de valeur visible
### Pourquoi c’est sensible
Les documents business recommandent “projet actif”.
Mais il faut encore vérifier que cette unité est :
- la plus lisible ;
- la plus rassurante ;
- la moins stressante ;
- la plus alignée avec la valeur.

### Impact si c’est faux
- pricing mal raconté ;
- charge mentale inutile ;
- comparaison défavorable avec alternatives.

### Ce qu’il faut tester
Comparer :
- projet actif ;
- dossier / export ;
- hybride ;
- éventuellement workspace, à titre de contrôle.

## Hypothèse 7 — L’abonnement mensuel peut tenir économiquement si l’usage est borné
### Pourquoi c’est sensible
Les unit economics indiquent :
- coût IA variable ;
- support plus lourd qu’un SaaS banal ;
- usage par pics ;
- risque d’un flat trop généreux.

### Impact si c’est faux
- mauvaise marge ;
- mauvais arbitrage entre petits et gros comptes ;
- besoin de reconfigurer vite le pricing.

### Ce qu’il faut tester
- temps moyen de remise au carré ;
- fréquence de revisites ;
- intensité des fonctions coûteuses ;
- volume de support par client actif ;
- différentiel mono-projet / multi-projets.

## Hypothèse 8 — La profondeur documentaire est perçue comme puissance, pas comme surcharge
### Pourquoi c’est sensible
Le projet vit précisément de sa profondeur.
Mais cette profondeur peut aussi devenir une objection.

### Impact si c’est faux
- friction d’usage ;
- support d’éducation élevé ;
- rejet par des builders qui veulent rester rapides.

### Ce qu’il faut tester
- niveau minimum jugé crédible ;
- niveau premium toléré ;
- format préféré : dossier, mémoire, checklist, export, autre.

## Hypothèse 9 — Le segment post-Lovable / post-builder bloqué est un vrai segment acheteur
### Pourquoi c’est sensible
Le segment est narrativement fort.
Mais il peut être plus séduisant intellectuellement que réellement monétisable.

### Impact si c’est faux
- mauvaise acquisition ;
- mauvais message ;
- mauvais ancrage du wedge.

### Ce qu’il faut tester
- entretiens directs ;
- frustrations récurrentes ;
- capacité réelle à payer ;
- alternatives déjà achetées ou non.

## Hypothèse 10 — Le produit peut rester software et ne pas dériver vers du service déguisé
### Pourquoi c’est sensible
Le produit doit aider beaucoup :
- cadrer ;
- clarifier ;
- reformuler ;
- garder la cohérence.

Cela peut faire monter le support.

### Impact si c’est faux
- marge trop faible ;
- dépendance forte au support humain ;
- inadéquation entre narration logiciel et réalité opérationnelle.

### Ce qu’il faut tester
- questions de support les plus fréquentes ;
- part du support liée à une incompréhension produit ;
- bornes d’usage et de guidance nécessaires.

## Priorisation des validations
### Priorité 1
- phrase de douleur principale ;
- wedge V1 exact ;
- objet vendu exact.

### Priorité 2
- moment d’entrée produit / moment de paiement ;
- niveau minimum de profondeur utile ;
- segment post-Lovable.

### Priorité 3
- fréquence de revisite ;
- intensité d’usage ;
- coût support ;
- corridor de prix acceptable.

## Synthèse finale
Le projet ne manque pas d’intuition stratégique.
Il manque surtout encore de preuves sur :
- ce qui fait acheter ;
- ce qui fait revenir ;
- et ce qui reste rentable.
