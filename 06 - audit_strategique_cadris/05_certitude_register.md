# Registre de certitude

## Confirmé
- Le produit ne doit pas être compris comme un simple app builder.
- La valeur centrale porte sur la structuration, la cohérence, la mémoire de décision et le dossier canonique.
- La cible prioritaire la plus crédible est le builder déjà initié aux outils IA de code, pas le novice complet.
- Le terrain de jeu le plus cohérent en V1 est le passage du proto / petite app vers un vrai SaaS plus sérieux.
- La douleur la plus crédible apparaît quand le projet grossit et qu’il n’existe plus de source de vérité fiable.
- En V1, utilisateur, acheteur et décideur sont souvent la même personne.
- Le produit paraît plus fort sur un projet déjà réel que sur l’idéation pure.
- La promesse crédible en V1 n’est pas “on gère tout le build”, mais “on met de l’ordre avant la casse totale, puis on garde une continuité légère”.
- Le scope V1 doit rester concentré sur des cas SaaS simples à moyens, structurables et transmissibles.
- La V1 privilégie avant-code fort + continuité légère pendant build.
- Le dossier canonique est la source de vérité prioritaire.
- Le modèle crédits appartient à l’historique du projet ; il n’est plus le modèle cible recommandé.
- L’unité de valeur retenue à ce stade est le projet actif.
- Le projet présente un niveau de cohérence suffisant pour continuer sous hypothèses.

## Hypothèses de travail
- Le produit sera acheté d’abord pour réduire la dérive et le risque de rebuild plus que pour “générer de la documentation”.
- Le livrable le plus monétisable au départ est un dossier canonique exportable servant de source de vérité.
- Le modèle économique le plus sain au départ ressemble à un logiciel ancré sur le projet actif, avec logique hybride compatible.
- La profondeur économique à moyen terme viendra de la continuité.
- Un sous-segment particulièrement prometteur est l’utilisateur post-Lovable / post-builder IA bloqué.
- Le meilleur wedge initial ressemble à un micro-SaaS B2B simple / mono-workflow avec squelette SaaS standard.
- Le point d’entrée principal le plus crédible est juste après le proto, quand la complexité commence à coûter.
- Une continuité légère suffit potentiellement à créer de la rétention.

### Impact
- ces hypothèses déterminent le wording de l’offre ;
- elles changent la structure tarifaire ;
- elles orientent le séquencement produit ;
- elles influencent directement la rentabilité potentielle ;
- elles conditionnent le cadrage du wedge V1.

### Pourquoi ces hypothèses ont été retenues
- elles sont cohérentes avec les JTBD centrés sur ordre, source de vérité et transmission ;
- elles collent aux contextes d’achat post-proto / post-chaos ;
- elles rendent le produit plus concret à vendre ;
- elles reflètent les documents business les plus récents ;
- elles réduisent le risque de présenter Cadris comme un builder de plus.

## Inconnus
- La phrase exacte de douleur d’achat qui convertit le mieux.
- Le moment précis du déclenchement de paiement.
- Le poids réel du besoin “dossier exportable” vs “mémoire continue”.
- Le type de SaaS initial à inclure explicitement dans la V1.
- Le niveau de sophistication documentaire perçu comme suffisamment puissant sans paraître lourd.
- Le corridor de prix acceptable.
- Le ratio réel entre usage ponctuel intense et usage continu léger.
- Le coût moyen par client actif.
- Le volume moyen de support par client actif.
- La capacité du segment post-Lovable à devenir un vrai segment acheteur.

### Pourquoi ce point reste inconnu
- validation terrain encore insuffisante ;
- pas encore assez de preuves de préférence nette ;
- l’économie récente du projet est mieux formulée que mesurée ;
- la fréquence réelle d’usage n’est pas encore observée.

### Quel impact potentiel
- mauvais packaging ;
- pricing mal ancré ;
- marge dégradée ;
- onboarding trop large ;
- rétention mal anticipée ;
- mauvais ciblage marketing ;
- scope trop ambitieux.

## Bloquants
- Définir la douleur d’achat principale en une phrase simple.
- Définir 3 cas inclus et 3 cas exclus pour le wedge V1.
- Trancher plus clairement la frontière avant-code vs pendant-build.
- Vérifier si le post-Lovable est un vrai wedge ou juste un angle séduisant.
- Identifier le niveau minimum de livrable jugé réellement actionnable.
- Mesurer rapidement la fréquence de revisite, l’intensité d’usage et la charge support.
- Éviter de fixer un pricing public définitif avant d’avoir sécurisé un minimum de signaux terrain.

### Pourquoi c'est bloquant
- sans cela, l’offre reste partiellement floue ;
- le pricing risque de raconter la mauvaise histoire ;
- le GPT suivant pourrait construire un messaging trop large ;
- le modèle récurrent pourrait être adopté sans preuve suffisante de fréquence et de marge.

### Ce qu'il faut obtenir pour débloquer
- entretiens ciblés ;
- tests de landing ;
- réactions à plusieurs formulations d’offre ;
- signaux de préférence sur l’unité de valeur ;
- mesures d’usage sur les parcours les plus coûteux ;
- premiers signaux de willingness to pay.

## Statut de transmission
- Transmission autorisée : **Oui sous hypothèses**
- Raison :
  - le noyau stratégique est suffisamment cohérent ;
  - le modèle économique est orienté dans une direction crédible ;
  - mais le pricing final, le wedge précis et la fréquence d’usage restent encore à valider.

## Handoff au GPT suivant
Le GPT suivant peut partir d’une base stratégique jugée cohérente mais incomplètement prouvée.
Il doit considérer comme bases fiables :
- builder post-proto ;
- source de vérité / dossier canonique ;
- projet actif comme unité de valeur ;
- continuité légère ;
- wedge SaaS simple à moyen ;
- avant-code fort.

Il doit traiter avec prudence :
- le corridor de prix ;
- l’abonnement mensuel pur ;
- la réalité de la rétention ;
- le segment post-Lovable comme wedge commercial principal.
