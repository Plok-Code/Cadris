# Contraintes initiales

## Stade du projet
Le projet est au stade **prototype**.

## Périmètre initial retenu
Le périmètre initial n'est pas "tout projet numérique" dès maintenant, mais un traitement **SaaS extrêmement complet**.  
L'intention exprimée est claire : pour le SaaS, le système doit aller très loin et couvrir un maximum de documentation, avec des ajouts selon les cas.

## Ressources humaines connues
L'équipe disponible comprend au moins :
- dev front ;
- dev back ;
- ML engineer ;
- data scientist ;
- data analyst ;
- data engineer ;
- sysops ;
- devops.

## Niveau technique du porteur
Le niveau technique déclaré est **avancé**.

## Budget et temps
- Budget : présenté comme **quasi illimité**
- Temps : **pas de limite claire exprimée à ce stade**

Ces points signifient surtout qu'il n'existe pas, pour l'instant, de contrainte forte de moyens.  
La vraie contrainte devient donc le **périmètre**, la **priorisation** et la **cohérence de construction**.

## Exigence de qualité
Le niveau d'ambition est très élevé :
- documentation très poussée ;
- qualité professionnelle ;
- cohérence transverse ;
- guidance d'exécution réelle ;
- couverture SaaS très large.

## Contraintes produit implicites
Même sans contrainte de budget ou de temps explicite, plusieurs contraintes structurantes existent déjà :
- ne pas laisser les LLM improviser les décisions critiques ;
- garder un fil cohérent entre les documents ;
- produire des documents exploitables par des humains et des LLM ;
- permettre une itération progressive du prototype vers V1 puis V2 ;
- conserver une vision du produit final visé par l'utilisateur.

## Contrainte de modèle économique
Le modèle économique retenu à ce stade repose sur :
- un système de crédits ;
- 10 crédits gratuits à l'inscription ;
- une consommation visible ;
- une alerte à 2 crédits ;
- à 0 crédit, pause ou mode brouillon non livrable ;
- rechargement requis pour obtenir le dossier final exportable.

## Contraintes encore non stabilisées
Les points suivants restent insuffisamment cadrés :
- prix des crédits ;
- taille moyenne d'un parcours SaaS complet ;
- niveau de personnalisation par sous-type de SaaS ;
- profondeur minimale obligatoire par document ;
- critères opérationnels de validation qualité.

## Lecture honnête de la situation
Le projet n'est pas réellement contraint par le manque de moyens déclaré.  
Il est surtout contraint par :
- la largeur du périmètre ;
- l'exigence de qualité ;
- la difficulté à industrialiser un dossier "très poussé" sans incohérences.

## Statut de fiabilité
### Confirmé
- Prototype.
- Équipe technique large.
- Niveau technique avancé.
- Ambition de couverture SaaS maximale.
- Modèle crédit retenu dans sa logique générale.

### Hypothèses de travail
- Les ressources annoncées seront réellement mobilisables.
- Le périmètre SaaS complet reste traitable de manière industrialisable.

### Inconnu
- Prix, coûts, temps moyen par dossier.
- Niveau de variabilité acceptable entre dossiers.

### Bloquant
- Aucun pour transmettre les fondations.
- Le GPT suivant devra vérifier les risques de surcharge de périmètre.
