# 02_pricing_hypotheses.md

## Principe general
Le pricing doit raconter une histoire simple :
**"Je paie pour faire travailler une equipe d'agents specialises sur mon projet et obtenir un cadrage executable."**

Le pricing ne doit pas raconter :
- "je paie pour generer du texte" ;
- "je paie pour quelques runs opaques" ;
- "je paie pour un simple document PDF" ;
- "je paie pour un concept abstrait de gouvernance".

## Structures de pricing possibles

### Option A - Paiement par mission
#### Description
Le client paie pour une mission de cadrage ou de recadrage sur un projet donne.

#### Pourquoi c'est plausible
- tres concret ;
- tres lisible ;
- bien aligne avec une logique "je soumets un sujet, les agents travaillent, je repars avec un vrai livrable".

#### Limites
- recurrence plus faible ;
- risque de sous-capter la valeur si le projet continue a vivre dans Cadris.

### Option B - Abonnement par projet actif
#### Description
Le client paie un abonnement tant qu'un projet actif est maintenu dans le systeme.

#### Pourquoi c'est plausible
- bon alignement avec la memoire et les revisions ;
- simple a comprendre pour une cible SaaS.

#### Limites
- moins bien aligne avec une valeur forte concentree sur une grosse mission initiale.

### Option C - Hybride recommande
#### Description
- une logique d'entree par mission ou par projet ;
- une logique de continuite pour revisions, nouvelles missions, supervision aval et enrichissement documentaire.

#### Pourquoi c'est plausible
- colle le mieux a la realite du produit ;
- permet une premiere vente tres concrete ;
- ouvre vers du recurrent sans brouiller la promesse.

## Hypothese principale recommandee

### Hypothese principale
**Hybride mission + continuite**

Exemple de lecture :
- entree : 1 mission de cadrage multi-agent incluse ;
- puis abonnement ou packs de missions pour revisions, pivots et build-review.

### Pourquoi cette hypothese est retenue
- le produit ressemble plus a une sequence de missions specialisees qu'a un simple espace de stockage ;
- le travail agentique a une valeur visible par mission ;
- la continuite a aussi une valeur credible si le projet evolue.

## Hypotheses alternatives

### Alternative 1
**Paiement par mission uniquement**
si la valeur principale percue reste le "grand cadrage initial".

### Alternative 2
**Abonnement simple**
si l'usage reel montre que les projets reviennent regulierement et que la profondeur se consomme dans le temps.

## Justification des choix

### Ce qu'il faut privilegier
- forte lisibilite ;
- lien direct avec la valeur percue ;
- packaging qui raconte la couverture agentique ;
- possibilite de faire monter en gamme sans jargon.

### Ce qu'il faut eviter
- credits visibles comme unite principale ;
- prix au nombre de tokens ou de runs ;
- prix qui laissent croire que l'utilisateur paie juste pour "de la doc" ;
- grille qui devient trop complexe trop tot.

## Zones encore speculatives
- willingness to pay exacte pour une mission de cadrage multi-agent ;
- tolerance a un prix premium si le dossier livre est vraiment dense ;
- elasticite selon le nombre de domaines couverts ;
- legitimite d'un palier dedie a la supervision du build.

## Orientation packaging en 3 niveaux maximum

### Niveau 1 - Core
- 1 mission de cadrage ;
- couverture des domaines essentiels ;
- dossier d'execution consolide ;
- export standard.

### Niveau 2 - Extended
- profondeur documentaire plus elevee ;
- plus de domaines ou de livrables ;
- revision ou pivot inclus ;
- export enrichi.

### Niveau 3 - Operating Layer
- plusieurs missions ;
- continuite de cadrage ;
- supervision build-review ;
- collaboration legere ;
- gouvernance plus robuste.
