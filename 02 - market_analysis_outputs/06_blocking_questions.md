# Questions bloquantes

## 1. Quel segment paiera vraiment en premier ?
### Pourquoi c’est bloquant
La cible actuelle “vibecodeurs débutants” est utile pour comprendre le problème, mais trop large pour un positionnement fiable.
Le produit peut résonner différemment selon :
- solo founder non-tech ;
- builder semi-tech ;
- agence ;
- PM/designer qui prototype ;
- petite équipe produit.

### Ce qu’il faut obtenir
- un segment d’entrée prioritaire ;
- un segment secondaire éventuel ;
- les critères qui rendent ce segment plus douloureux et plus acheteur.

## 2. Quelle douleur domine réellement à l’achat ?
### Pourquoi c’est bloquant
Le brief mélange plusieurs douleurs réelles :
- mauvaise stack ;
- incohérence documentaire ;
- perte de temps ;
- perte de confiance ;
- difficulté à piloter le build ;
- redémarrage coûteux.

Sans hiérarchie claire, la proposition de valeur risque de rester trop large.

### Ce qu’il faut obtenir
- la douleur dominante ;
- les douleurs secondaires ;
- la formulation d’achat la plus naturelle pour l’utilisateur.

## 3. Quel est le bon wedge initial dans le périmètre SaaS ?
### Pourquoi c’est bloquant
“SaaS à 100%” est une ambition utile, mais trop large pour prouver rapidement une qualité très élevée.
Sans wedge, le projet risque de devenir une usine à cas particuliers.

### Ce qu’il faut obtenir
- un sous-périmètre initial homogène ;
- les types de SaaS explicitement inclus ;
- les types de SaaS exclus temporairement.

## 4. Quel livrable déclenche la valeur perçue immédiate ?
### Pourquoi c’est bloquant
Le projet peut produire beaucoup de valeur potentielle, mais il faut savoir quel artefact ou résultat fait dire à l’utilisateur :
“oui, ceci m’aide réellement”.

### Ce qu’il faut obtenir
- le premier livrable décisif ;
- le moment où l’utilisateur ressent la valeur ;
- les preuves visibles de qualité et de cohérence.

## 5. Le produit doit-il être autonome ou s’insérer dans des outils existants ?
### Pourquoi c’est bloquant
Le marché dispose déjà de builders IA, de docs, de PRD tools et de workflows établis.
Si le projet essaie de tout remplacer, le coût d’adoption peut être trop élevé.

### Ce qu’il faut obtenir
- une hypothèse claire :
  - produit principal ;
  - surcouche ;
  - plugin ;
  - copilote ;
- les outils avec lesquels il doit coexister au départ.

## 6. Jusqu’où l’aval doit-il aller ?
### Pourquoi c’est bloquant
Le contrôle aval est potentiellement très différenciant, mais il peut aussi démultiplier la complexité.
Il faut savoir si l’aval :
- suggère seulement le prochain prompt ;
- vérifie la cohérence ;
- bloque certaines dérives ;
- ou devient presque un superviseur d’agent.

### Ce qu’il faut obtenir
- le rôle exact de l’aval ;
- ses limites ;
- les cas où il alerte ;
- les cas où il impose une vérification.

## 7. Comment prouver que la profondeur documentaire n’est pas une surcharge ?
### Pourquoi c’est bloquant
Le projet repose sur une documentation très poussée.
Or le marché peut aimer la vitesse plus que la profondeur.

### Ce qu’il faut obtenir
- une preuve que la profondeur réduit réellement le risque ;
- une forme de restitution lisible ;
- un niveau de détail par défaut acceptable.
